# ADR-0010: Authentication & Authorization

- **Status:** Accepted
- **Date:** 2026-07-06
- **Deciders:** Platform team
- **Related:** [ADR-0001](0001-language-and-runtime.md), [ADR-0006](0006-temporal.md), [ADR-0009](0009-api-gateway.md), [ADR-0017](0017-url-and-domain-structure.md)

## Context

The platform supports:

- **B2C users** — public sign-up, email/password, social login, MFA, recovery.
- **B2B organisations** — multi-tenant orgs, SSO (OIDC/SAML), per-org roles, SCIM provisioning.
- **Service-to-service identity** — internal calls trusted by network policy ([ADR-0003](0003-cluster-topology.md)); identity headers are forwarded, no per-call token.
- **Third-party API consumers** — external developers obtaining tokens to call our public APIs.

Constraints inherited from earlier ADRs:

- **Self-host only.** No Auth0/Clerk/WorkOS.
- **Per-service cost matters.** JVM-based stacks pay a footprint tax incompatible with [ADR-0001](0001-language-and-runtime.md).
- **The edge validates tokens; services decide permissions.** Set in [ADR-0009](0009-api-gateway.md).

Hard requirement on the UI:

- **The login UI is a custom Next.js implementation of a Figma design.** It is not the IdP's hosted pages, themed templates, or a forked vendor UI. The IdP must expose a headless, API-first identity flow that Next.js drives end-to-end.

Authorization shape:

- Day-one features include cases beyond flat RBAC (resource ownership across orgs, shared resources, role-per-org-per-resource).
- Temporal is available ([ADR-0006](0006-temporal.md)), which makes the dual-write between application DB and authz store a routine workflow pattern.

The decision splits into:

1. **Identity Provider** — login, registration, sessions, user tokens.
2. **OAuth2 / OIDC authorization server** — machine and third-party tokens.
3. **Authorization engine** — given an authenticated principal, may they perform an action.

## Decision drivers

1. **Headless IdP** — required by the Figma constraint.
2. **Polyglot-tolerant integration contracts** — auth is a *protocol* contract (token format, header names, validation rules), expressible in any language. Per [ADR-0001](0001-language-and-runtime.md)'s escape hatches.
3. **Files in the repo** — identity schemas, OAuth2 client definitions, authz schemas all version-controlled.
4. **Operational shape consistent with the rest of the platform** — Go binaries on Postgres, not JVM.

## Considered options

### Identity Provider

- **Ory Kratos** — fully headless self-service flows; identity schemas as JSON Schema files; YAML config; Go codebase; Apache 2.0. The reference implementation of the headless pattern.
- **Zitadel** — single Go binary, native multi-tenancy; custom login is supported but is the road less travelled, and hosted pages are the design centre.
- **Keycloak** — most mature OSS IdP; comprehensive feature set; JVM footprint, custom UI via Authentication Sessions API is a minority path, extension work in Java.
- **Authentik** — Python/Django + Celery + Redis; B2C self-service flows less mature than Kratos.

### OAuth2 / OIDC server

- **Ory Hydra** — Go, Postgres, OAuth2/OIDC server, headless; integrates with Kratos for user-consent flows.
- Roll our own — explicitly rejected ([ADR-0000](0000-platform-foundations.md): no "temporary" infrastructure).

### Authorization engine

The day-one requirements are ReBAC, not flat RBAC: cross-org sharing, role-per-org-per-resource, and reverse-index questions ("which resources can this user see"). That narrows the field to Zanzibar-style engines, compared below on what decides it — capability, governance and longevity, and fit for a Go/Postgres central-service platform. The trade-offs are qualitative, so this is prose, not a scoring grid.

| Engine | Model & capability | Governance & longevity | Fit for this platform | Verdict |
| --- | --- | --- | --- | --- |
| **OpenFGA** *(chosen)* | Zanzibar ReBAC; ABAC via CEL conditions + contextual tuples; reverse-index (`ListObjects`/`ListUsers`) | **CNCF** — vendor-neutral, multi-vendor contributors | Single Go binary on Postgres; `.fga` DSL + `fga model test` in CI + `fga` CLI; official Go SDK behind the `Checker` seam | **Chosen.** Matches SpiceDB on capability and operational shape and wins on vendor-neutral governance. Contextual tuples also let a check read data at request time, easing the DB↔authz sync the dual-write carries. |
| **SpiceDB** | Zanzibar ReBAC; ABAC via CEL caveats; strongest reverse-index + `Watch` APIs; `ZedToken` consistency | Single-vendor (Authzed) | Single Go binary on Postgres; `.zed` schema + `zed` CLI | The credible fallback — equal on capability and the reference for reverse-index/`Watch`. Loses only on governance (single-vendor vs CNCF); its `ZedToken` consistency edge bites only at very large scale (see *Accepted limitation* below). |
| **Ory Keto** | Zanzibar ReBAC; ABAC only via OPL (awkward); no conditions/`Watch`/batch | Ory — single-vendor; some features OEL-licensed | Fits the existing Ory stack (Kratos/Hydra) | Thinnest feature set of the mature engines and OEL-gated extras, with no offsetting advantage over OpenFGA. |
| **Permify** | ReBAC + first-class ABAC (attributes + rules); native multi-tenancy | Single-vendor, recently acquired (FusionAuth); smaller community | Go binary on Postgres | Capable, but narrower adoption and just-changed ownership make it a weaker long-term bet than OpenFGA or SpiceDB. |
| **Topaz** | ReBAC directory **plus full OPA/Rego** policy — most expressive; but weak reverse-index / large-scale query APIs | Sponsor (Aserto) wound down; community-maintained; pre-1.0 | Edge/sidecar with an embedded directory — must replicate authz data to every instance | Richest policy language, but pre-1.0 with no vendor, the worst DB↔authz data-sync of the field, and lock-in to proprietary Rego. Too much operational risk for a platform baseline. |
| **Casbin** | PERM-model **library** (RBAC/ABAC); role-inheritance only — **no relationship-graph traversal or reverse index** | Apache (incubating) — vendor-neutral | In-process library; polyglot ports must agree on matcher semantics | Not a Zanzibar/ReBAC engine, so it is weakest at the exact day-one requirements. As a library it *is* the "in-process RBAC per service" option below. |

OpenFGA and SpiceDB are the two mature, capable options; the deciding factor is OpenFGA's CNCF vendor-neutral governance, with SpiceDB the ready fallback if that ever ceases to matter. Any two Zanzibar engines are interchangeable at the architecture level: the `Checker`/`Granter` seam in `libs/go/authz/` is engine-agnostic, so reopening this decision touches one library, the model file, and the chart — not the services.

- **In-process RBAC per service** — the non-Zanzibar alternative. Rejected: it diverges per service; the "occasional complex case" already exists from day one (cross-org sharing, role-per-resource); migrating to a Zanzibar engine later means a per-feature data migration.

## Decisions

### Identity Provider: Ory Kratos

Kratos handles human authentication: registration, login, MFA, recovery, account settings, social login. Identity schemas as JSON Schema files at `infra/auth/kratos/identity-schemas/`. All other config in YAML at `infra/auth/kratos/`. Deployed via Helm at `infra/helm/platform/ory/`.

The Next.js login UI lives in `apps/frontend/src/app/(landing)/auth/` and drives Kratos self-service flows directly.

The Kratos session cookie is configured `SameSite=Lax` + `Secure` + `HttpOnly` in `infra/auth/kratos/`. This is the
first line of CSRF defence (it blocks the classic cross-site POST); Kratos's built-in anti-CSRF token covers the
self-service flows themselves, and the edge Origin check ([ADR-0009](0009-api-gateway.md)) backstops other
cookie-authenticated mutations. The browser-side CSP and Server-Actions CSRF handling are [ADR-0014](0014-frontend.md)'s.

### OAuth2 / OIDC server: Ory Hydra, when a public API exists

Hydra is the OAuth2/OIDC server, deployed when a project exposes a public API or external machine clients (the
`hydra_thirdparty` flag, [ADR-0009](0009-api-gateway.md)). It issues **authorization-code and client-credentials tokens
for third-party developers and external machines.** Those tokens are validated at the edge by Oathkeeper and converted
to the standard identity headers ([ADR-0009](0009-api-gateway.md)), so the internal request shape is identical whether
or not Hydra is deployed.

Hydra is **not** used for internal service-to-service calls — those carry no token and are gated by Cilium NetworkPolicy
([ADR-0003](0003-cluster-topology.md)). When deployed, Hydra is the one and only issuer of external tokens; no service
mints its own.

Hydra config lives at `infra/auth/hydra/`. Third-party OAuth2 clients are declared in YAML and applied by a Hydra CLI
invocation in a Helm post-sync hook.

### B2B Organisations: an `orgs` service on top of Kratos

Kratos stores identities, not orgs. Multi-tenancy is implemented as a first-party `services/orgs/` service that owns:

- Organisations and memberships.
- The "active organisation" for a user session (set by a post-login Kratos webhook).
- Org-level invitations and SSO connection records.

The `orgs` service is built ahead of the first B2B feature.

#### Tenancy model: a personal org per user, created eagerly at registration

An organisation is the unit of authorization — every protected resource in `model.fga` belongs to an `org`,
and a user only ever acts *through* a role in some org (`org->read`, `org->write`). There is no notion of a
resource owned by a bare user with no org. A user with zero orgs is therefore an account that can own nothing
and do nothing.

So the template makes membership a day-one invariant: **the moment a Kratos identity is created, it gets a
personal organisation (a "tenant of one") in which it is `admin`.** The Kratos post-registration webhook starts
the `RegisterUser` Temporal workflow, whose two activities are the same dual-write as any authz mutation — the
`orgs`-DB row (`orgs` + `org_members`) and the OpenFGA `org:<id>#admin@user:<id>` tuple. Because `org_members`
is many-to-many, a user can then create further orgs and be invited into others; the personal org is simply
their first tenant, never a ceiling.

This is deliberately the **most flexible** model, not the most minimal, and a reader should choose with both in
view:

- **Why eager personal-org.** It holds the invariant "every user is in ≥1 org" with **zero branching** — no
  code path anywhere has to handle a user who has no tenant yet. It is a *superset*: a sales-led "sign up →
  create a company → invite staff" flow layers on top unchanged, because the data model (M:N users↔orgs) already
  allows it. And it keeps each new user isolated in their own tenant — a *shared* default org would put unrelated
  users in one tenant and break isolation, so it is explicitly rejected.
- **What it costs.** A clutter of one-person orgs. Presentation, not model: a UI may hide personal orgs from the
  org switcher until the user creates or joins a second one.

The personal org's default `name` is a **generic constant** (`"Personal workspace"`), not the user's email. An org
is a tenant that may later hold a whole team, and its `name` is shown to every member the user invites — an email
there would leak PII into a shared, display-facing field ([ADR-0023](0023-data-lifecycle-privacy.md)) and would go
stale when the mutable email changes. The stable identity is the org `id` (a UUID); `name` exists to be
human-readable, so a random string would be strictly worse than a generic label. The user renames it from the
console.

**Alternatives, and how to change it.** The choice lives entirely in the registration webhook path
(`services/orgs/internal/{handlers,workflows,activities}`) — the rest of the platform only assumes the invariant,
not the mechanism:

| Model | Tenant created | Gain vs. eager | Cost |
| --- | --- | --- | --- |
| **Personal-org, eager** *(chosen)* | Auto, at registration | Zero org-less branching; superset of the others | One-person-org clutter; email-named default reads as a user clone |
| **Lazy tenant** | On first action that needs one | Less clutter | Every caller must handle the org-less user; "N identities, 0 orgs" becomes normal, not an edge case |
| **Owner-creates-first** (explicit B2B) | Manually, as a deliberate step | Cleanest for pure B2B | A freshly-signed-up user sits in limbo until they create/join one |
| **Invite-only / SCIM** | By an org admin or IdP; no open signup | Fits enterprise/regulated | No self-serve growth; provisioning path must be built up front |
| **No org** (pure B2C) | Never — user *is* the boundary | Simplest if teams never come | Re-adding tenancy later is a data migration; needs `model.fga` ownership relations reworked |

Switching model means editing this webhook path and, for the B2C case, the `model.fga` ownership relations; the
`Checker` seam and every service's authz call stay identical.

### Authorization Engine: OpenFGA

OpenFGA is the authorization engine, accessed through a shared `authz.Checker` interface in `libs/go/authz/`. Services never call the OpenFGA SDK directly (a depguard rule in `.golangci.yml` confines the SDK to `libs/go/authz/`).

One engine serves every project, at whatever complexity it needs: **L1 (role per org, which *is* flat RBAC) → L2 (resource ownership) → L3 (sharing / hierarchy)**. The `Checker` interface is identical at every level; only `model.fga` grows, so a simple instance starts at an ~15-line L1 model and most stay there forever, while nobody ever migrates tools. L1 is the first-class default, not a fallback. Full rationale and copy-paste models: [docs/auth/authorization-levels.md](../auth/authorization-levels.md).

- The model is authored in the OpenFGA **DSL** at `infra/auth/openfga/model.fga` (the single source of truth) and transformed to `model.json` for the API by `mise run gen:authz-model`. **One global model**, not per-service, because authz relationships routinely cross service boundaries.
- Check assertions live at `infra/auth/openfga/fga.yaml`. CI runs `fga model test` (and asserts `model.json` is in sync with the DSL) on every PR, via `mise run lint:authz`.
- Deployed via Helm at `infra/helm/platform/openfga/`, sharing the platform Postgres.
- **Deployed directly (Deployment + Service), not via an operator.** OpenFGA ships no first-party operator; we run an `openfga migrate` initContainer before `run` — the same pattern our own services use for their migrations. A phased datastore migration on a major upgrade is handled manually per that release's notes.
- **Store + model bootstrap.** Unlike a single mutable schema, OpenFGA needs a **store** (a ULID) and an **authorization model** (immutable, versioned) created in-cluster. A seed Job (`infra/helm/platform/openfga/templates/seed.yaml`) ensures a store named `platform` exists (list → find-by-name → else create; idempotent across re-syncs), writes the model, and applies the static ops grants over the HTTP API. Services resolve the store by that name at first use (`libs/go/authz/`), so no store-ID plumbing is needed; `store_id`/`model_id` are opaque, deploy-time values, not secrets. A check omits the model ID and uses the store's latest model.

### Accepted limitation: no consistency token

SpiceDB's `ZedToken` gives per-request read-after-write consistency; OpenFGA has **no equivalent** ([openfga/roadmap#67]), offering only a coarse `HIGHER_CONSISTENCY` flag. This is **not a day-one concern** — it only bites at very large, heavily-replicated scale, and the template does not depend on it (the `Checker` runs at default consistency and the dual-write does not thread a token). Revisit only if an instance grows huge *and* roadmap#67 is still open; the lever then is `HIGHER_CONSISTENCY` on the reads that need it.

[openfga/roadmap#67]: https://github.com/openfga/roadmap/issues/67

### Dual-write discipline

Every authz-relevant mutation (create / share / transfer / delete on a protected resource) is wrapped in a Temporal workflow ([ADR-0006](0006-temporal.md)). The workflow contains both the application-DB write and the OpenFGA write as activities, so the pair cannot half-apply. Reads run at default consistency (see the limitation above) — no consistency token is threaded.

Direct application-DB writes that mutate authz-relevant tables outside a workflow are a review-blocker enforced by a `depguard`-style lint rule against the relevant store methods.

### Identity: validated at the edge, carried as headers

Tokens are validated once, at the edge ([ADR-0009](0009-api-gateway.md)). Past the edge there are **no tokens** —
identity travels as a fixed set of trusted headers, the same shape for every request.

- **Edge validation.** Ory Oathkeeper validates the Kratos session cookie (browser) or the Hydra-issued JWT
  (third-party / machine clients). JWTs are RS256 with keys at a JWKS endpoint; validation requires issuer, audience,
  expiry, signature, and `nbf`, with 30 s clock-skew tolerance, defined once in `docs/auth/jwt-validation.md`.
- **Identity headers.** Oathkeeper strips any client-supplied identity headers and injects authoritative `X-User-Id`,
  `X-Org-Id`, `X-Roles` ([ADR-0009](0009-api-gateway.md)). The underlying claim set is `sub`, `org_id` (active
  organisation), `roles` (org-scoped), plus standard timing claims.
- **Services read headers.** The Go reference reader is `libs/go/authmw/`; it reads the identity headers into a typed
  principal. Services never fetch JWKS or parse a JWT. Escape-hatch languages implement the same header contract.
- **Service-to-service calls** carry no token. They forward the same identity headers and are gated by Cilium
  NetworkPolicy ([ADR-0003](0003-cluster-topology.md)); only sanctioned callers can reach a service's port. This is the
  platform's internal trust boundary — network identity, not a per-call machine token.

A small **conformance test suite** at `tools/auth-conformance/` ships with the repo: identity-header fixtures with
expected principals and authorization outcomes. Any non-Go service must pass it.

### Authorizing operator tooling at the edge

"The edge validates, services decide" assumes a first-party service exists to make the decision. Operator dashboards —
Hubble UI, Grafana, Argo CD, Temporal, the MinIO console, the Lowdefy console — are third-party and cannot call `Checker`,
so their authorization happens **at the edge**, in two layers ([ADR-0017](0017-url-and-domain-structure.md)):

- **Coarse gate (mandatory) — a claim, not a `Checker` call.** The ops-tier Oathkeeper requires the caller's `X-Roles`
  to contain `operator` (from the `operator` trait on the Kratos identity schema) plus an **AAL2** session. This reads
  only the session and its claims and makes **no OpenFGA call** — deliberately, so the debugging surface does not share
  fate with the product authorization plane: an OpenFGA outage must not lock every operator out of the dashboards they
  need to diagnose it. See [docs/ops/break-glass.md](../ops/break-glass.md).
- **Fine gate (optional) — per-tool `remote_json` → `Checker`.** When per-tool grants are wanted, the ops route adds the
  `remote_json` authorizer calling `Checker`, modelling each tool as a `dashboard` resource. For 3–8 operators this is
  typically deferrable, so the `dashboard` resource is optional day-one.

This is the only sanctioned edge-side permission decision; product surfaces continue to decide in-app/in-service through
`Checker`. The ops trust tier, hostnames, session isolation, and the `dashboard` resource shape are specified in
[ADR-0017](0017-url-and-domain-structure.md).

### Day-one component cost (summary)

The internal-only auth stack at launch is:

- **Kratos** (Helm) — human login.
- **Oathkeeper** (Helm) — edge validation + identity-header injection ([ADR-0009](0009-api-gateway.md)).
- **OpenFGA** (Helm) — authorization.
- **`services/orgs/`** (first-party service).
- **`libs/go/authmw/`** (header reader) + **`libs/go/authz/`** (`Checker`) libraries.

**Hydra is deployed only when a project exposes a public API or external machine clients** (`hydra_thirdparty` flag,
[ADR-0009](0009-api-gateway.md)); internal-only projects do not run it. There is no service-to-service token: internal
trust is Cilium NetworkPolicy ([ADR-0003](0003-cluster-topology.md)). All components share the platform Postgres and the
platform observability stack.

## Consequences

### Positive

- Login UI freedom. The Figma design is implemented exactly, with no fight against vendor templates.
- One way to obtain a token for human or machine identity, day one and forever.
- Authz is a single coherent model from day one — no deferred migration, no per-service RBAC tables.
- Tooling supports the work: schema in files, tests in files, CLI for debugging, GitOps for everything.
- Polyglot-safe. Auth integration is a protocol contract; Rust or Python services participate on equal terms.
- Dual-write risk is structurally contained by routing authz-relevant mutations through Temporal.

### Negative / Risks

- **Three new platform components on day one for internal-only projects** (Kratos, Oathkeeper, OpenFGA) plus the `orgs` service; Hydra is added only when a public API exists. Accepted; each is best-in-class for its job, and the constraints (headless login from day one, ReBAC from day one) make consolidation impossible without compromise. Dropping the service-to-service token in favour of Cilium NetworkPolicy removes a per-call validation path from every service.
- **Building B2B organisations on top of Kratos** is real work. Accepted; the alternative (a bundled-orgs IdP) costs the Figma requirement.
- **Dual-write discipline must be enforced.** A direct DB write that skips the Temporal workflow is a silent authz bug. Mitigated by lint rules, code-review checklist, and integration tests asserting OpenFGA state after every workflow.

### Follow-ups

- `infra/helm/platform/ory/` (Kratos + Oathkeeper, plus Hydra when `hydra_thirdparty` is set) and `infra/helm/platform/openfga/`.
- `infra/auth/oathkeeper/` access rules (validate session/JWT → strip → inject identity headers).
- `libs/go/authmw/` (JWT middleware) and `libs/go/authz/` (`Checker` interface + OpenFGA-backed implementation).
- `infra/auth/kratos/`, `infra/auth/hydra/`, `infra/auth/openfga/`.
- `services/orgs/` scaffold.
- `tools/auth-conformance/` conformance suite.
- `docs/auth/jwt-validation.md` rules.
- `docs/auth/dual-write.md` documenting the Temporal pattern for authz-relevant mutations.

## Rules

- Human identity is owned by Ory Kratos. There is no alternate user store.
- External tokens (third-party and machine clients) are issued by Ory Hydra, which is deployed only for projects exposing a public API (`hydra_thirdparty` flag, [ADR-0009](0009-api-gateway.md)). No service issues its own tokens.
- The login UI is the Next.js app driving Kratos self-service flows. Kratos hosted pages are not used.
- The Kratos session cookie is `SameSite=Lax` + `Secure` + `HttpOnly`. CSRF defence is layered: SameSite cookies, Kratos's built-in token on auth flows, Next.js Server-Actions Origin checks ([ADR-0014](0014-frontend.md)), and the edge Origin check on other cookie-authenticated mutations ([ADR-0009](0009-api-gateway.md)).
- B2B organisations are owned by `services/orgs/`. Other services consult it via its HTTP API.
- The organisation is the unit of authorization: every protected resource belongs to an `org`, and a user acts only through a role in an org. Every Kratos identity gets a personal org (`admin`) at registration via the `RegisterUser` dual-write, so "every user is in ≥1 org" is an invariant no code path may assume away. A single shared default org is forbidden — it breaks tenant isolation. Changing the tenancy model (lazy, owner-first, invite-only, or no-org) is confined to the registration webhook path in `services/orgs/` and is a deliberate per-project decision, not a code-review default.
- Authorization is OpenFGA, accessed only through `libs/go/authz/`'s `Checker` interface. Direct OpenFGA SDK use outside this library is forbidden.
- The OpenFGA schema is a single global file at `infra/auth/openfga/model.fga`. Per-service schemas are forbidden.
- Authz-relevant mutations run inside a Temporal workflow with the DB write and the OpenFGA write as separate activities. Direct DB writes to authz-relevant tables outside a workflow are forbidden.
- Inline role checks in handlers are forbidden. Every permission decision goes through `Checker.Allowed(ctx, action, resource)`.
- Third-party operator dashboards are gated at the edge by a **coarse claim**: `X-Roles` contains `operator` (the `operator` trait on the Kratos identity) plus an **AAL2** session, enforced by the ops-tier Oathkeeper with **no OpenFGA call**, so the debugging surface never shares fate with the product authz plane ([ADR-0017](0017-url-and-domain-structure.md), [docs/ops/break-glass.md](../ops/break-glass.md)). Optional per-tool refinement adds the `remote_json` authorizer → `Checker` (`dashboard:<tool>#view`). Product surfaces still decide in-app/in-service through `Checker`.
- Simple instances use an L1 flat-RBAC schema (role per org); this is the first-class default, not a fallback. L2/L3 grow the same schema on the same engine, never a tool migration ([docs/auth/authorization-levels.md](../auth/authorization-levels.md)).
- Tokens are validated once at the edge by Oathkeeper ([ADR-0009](0009-api-gateway.md)); services do not validate tokens. JWTs are RS256 with keys at JWKS; validation rules are defined in `docs/auth/jwt-validation.md`.
- Identity is carried platform-wide as `X-User-Id`, `X-Org-Id`, `X-Roles` headers, injected by Oathkeeper at the edge and forwarded unchanged on internal calls. Services read identity only from these headers and never parse a token. Renaming the headers requires editing [ADR-0009](0009-api-gateway.md) and this ADR.
- Service-to-service calls carry no token; they forward the identity headers and are gated by Cilium NetworkPolicy ([ADR-0003](0003-cluster-topology.md)). Shared secrets, HMAC schemes, and per-call machine tokens are forbidden.
- Non-Go services read the identity headers via the same contract and pass `tools/auth-conformance/` before merging.
- Auth configuration (Kratos config, Oathkeeper config + access rules, Hydra clients, Kratos identity schemas, the OpenFGA `.fga` model) is canonical **only** at `infra/auth/*`. It is delivered to the charts by injecting each file into the relevant value — Helm `--set-file` locally and ArgoCD `source.helm.fileParameters` in deployed environments — never hand-copied inline into chart `values.yaml`. CI fails on a re-inlined copy. (Ory runs as a dependency subchart, so `.Files.Get` cannot populate its values; file injection is the single-source mechanism.)
